# Minecraft 多服务器模组管理工具 — 设计规格说明

## 1. 概述

### 1.1 项目目标
开发一款 Windows 桌面应用，用于统一管理多个 Minecraft 服务端（Forge/NeoForge）的模组及其他目录文件。支持公共模板 + 服务器覆盖的同步模式，集成 Modrinth 模组搜索、下载与更新功能。

### 1.2 核心需求
- 管理 **4 个** Minecraft 服务端目录（可扩展）
- 支持 `mods`、`config`、`kubejs`、`tacz` 等子目录的管理
- 公共模板 + 服务器专属覆盖的同步机制
- 模组浏览、启用/禁用、删除、Modrinth 下载、版本检测与更新
- 服务器别名自定义

## 2. 技术选型

| 层级 | 技术 | 说明 |
|------|------|------|
| 桌面框架 | Electron | 跨平台兼容，文件系统 API 丰富 |
| 前端框架 | Vue 3 + Vite | 现代响应式 UI |
| UI 组件库 | Element Plus | 完善的中文组件库 |
| 后端逻辑 | Node.js (Electron 主进程) | 文件操作、API 调用 |
| 本地存储 | better-sqlite3 (SQLite) | 配置与状态持久化，支持结构化查询和并发读写 |
| 模组源 | Modrinth API | 搜索、查询、下载无需 Token |

### 3.2 模块职责

| 模块 | 职责 |
|------|------|
| **文件系统管理** | 读取目录结构、文件操作（复制、移动、删除、重命名）、监听文件变更、文件哈希比对 |
| **Modrinth API 客户端** | 搜索模组、查询版本信息、下载模组文件（含请求队列与指数退避应对限流） |
| **同步引擎** | 公共模板与服务器目录的差异比对（基于文件哈希）、文件同步、冲突检测、覆盖关系管理 |
| **配置管理** | 服务器路径、别名、排除规则等配置的持久化与读取 |

## 4. 数据模型

### 4.1 配置文件结构 (config.json)

```json
{
  "templatePath": "D:/mc-template",
  "templateExcludedDirs": ["logs", "crash-reports", "cache"],
  "templateExcludedFiles": [".DS_Store", "thumbs.db"],
  "servers": [
    {
      "id": "server-1",
      "name": "生存服",
      "path": "D:/servers/survival",
      "excludedDirs": ["logs", "crash-reports"],
      "excludedFiles": ["servers.dat", "usercache.json"]
    },
    {
      "id": "server-2",
      "name": "工业服",
      "path": "D:/servers/industrial",
      "excludedDirs": ["logs"],
      "excludedFiles": []
    }
  ]
}
```

### 4.2 模组缓存数据 (mods-cache.json)

```json
{
  "servers": {
    "server-1": {
      "mods": [
        {
          "fileName": "sodium-0.6.0.jar",
          "modId": "sodium",
          "displayName": "Sodium",
          "version": "0.6.0",
          "enabled": true,
          "source": "template",
          "modrinthId": "AANobbMI",
          "latestVersion": "0.6.1",
          "hasUpdate": true
        }
      ]
    }
  }
}
```

### 4.3 文件覆盖标记

文件在服务器与模板之间的覆盖关系通过 `overrides.json` 元数据文件记录，路径为 `{templatePath}/.manager/overrides.json`：

```json
{
  "server-1": {
    "overriddenFiles": [
      "config/special.toml"
    ],
    "serverOnlyFiles": [
      "mods/extra-mod.jar",
      "tacz/server1_pack/"
    ]
  }
}
```

规则：
- **继承自模板**：服务器中有相同路径的文件，且不在 `overriddenFiles` 列表中 → 跟随模板更新
- **已覆盖 (overridden)**：服务器中的文件版本优先级高于模板，模板更新时不覆盖此文件
- **服务器独有 (server-only)**：模板中不存在的文件和目录，仅在该服务器存在
- **覆盖判断依据**：通过 SHA256 哈希比对判断文件是否被用户修改。与模板哈希一致 = 未覆盖；不一致 = 已覆盖

### 5.2 模组管理界面

| 功能 | 描述 |
|------|------|
| 模组列表 | 显示文件名、版本、状态（启用/禁用/可更新）、来源（模板/专属）|
| 搜索筛选 | 按名称搜索、按来源（template/server-only）筛选、按状态（enabled/disabled/hasUpdate）筛选、按名称/时间排序 |
| 启用/禁用 | 通过重命名 `.jar` → `.jar.disabled` 实现禁用，不删除文件 |
| 删除模组 | 直接从 mods 目录删除文件 |
| 从 Modrinth 安装 | 搜索 Modrinth，选择版本，下载到指定服务器的 mods 目录 |
| 版本检测 | 通过 Modrinth API 查询最新版本，标记可更新项 |
| 模组更新 | 删除旧文件 → 下载新版本，更新后询问是否同步到其他服务器 |
| 空状态 | 模组列表为空时显示引导提示："暂无模组，点击 + 从 Modrinth 安装" |

### 5.3 设置页面

| 设置项 | 说明 |
|--------|------|
| 公共模板路径 | 设置/修改公共模板所在目录 |
| 服务器管理 | 添加/删除/修改服务器路径、别名、排序 |
| 排除规则 | 全局排除规则（对所有服务器生效）+ 各服务器独立排除规则 |
| 启动时检查更新 | 是否在软件启动时自动检查模组更新 |
| 关于 | 软件版本、开源协议等 |

### 5.4 同步管理界面

| 功能 | 描述 |
|------|------|
| 差异检测 | 比对公共模板与各服务器目录（基于 SHA256 哈希），列出差异文件 |
| 同步推送（模板 → 服务器） | 将模板文件推送到所选服务器 |
| 同步收集（服务器 → 模板） | 将某服务器的新增/修改文件同步回公共模板，再推送到其他服务器 |
| 冲突处理 | 当模板和服务器同时修改同一文件时，显示对比让用户选择保留版本 |
| 覆盖管理 | 标记/取消标记某文件为服务器专属覆盖 |
| 操作日志 | 记录每次同步操作的时间、文件列表、结果（成功/失败），支持查阅和重试 |

## 6. 核心交互流程

### 6.1 模组更新流程

```
用户点击「更新」按钮
  → 检查模组文件是否被占用（被 Java 进程锁定？）
     ├── 被占用 → 提示"请先关闭服务器"并中止
     └── 未被占用 → 继续
  → 删除当前服务器中的旧 .jar 文件
     ├── 删除失败 → 报错并回滚，保留旧文件
     └── 删除成功 → 继续
  → 从 Modrinth 下载最新版本
     ├── 网络错误 → 捕获异常，提示重试，保留旧文件
     └── 下载成功 → 写入 mods 目录
  → 弹窗显示该模组在其他服务器中的分布情况
     ├── 用户勾选需要同步的服务器（含公共模板）
     ├── 点击「同步到所选服务器」
     │   → 逐个替换其他服务器的旧文件
     │   ├── 全部成功 → 显示成功摘要
     │   └── 部分失败 → 显示失败列表，提供重试入口
     └── 点击「仅更新当前」→ 只更新当前服务器
```

### 6.2 同步流程

```
用户进入同步面板
  → 系统比对公共模板与所有服务器目录（SHA256 哈希）
  → 列出差异项，分类展示：
     ├── 📄 模板新增 → 将推送到服务器
     ├── 📄 模板修改 → 将推送到服务器（除非已覆盖）
     ├── 📄 模板删除 → 将同步删除
     ├── 📄 服务器新增 → 可收集回模板（反向同步）
     └── ⚠️ 冲突 → 模板和服务器端均有修改

  → 用户处理冲突（若有）：
     ├── 文本文件（.toml, .json, .js）：按行 diff 对比，选择保留版本
     └── 二进制文件（.jar, .zip）：文件级别选择保留模板版或服务器版

  → 用户确认操作后执行同步
     ├── 同步成功 → 刷新状态，记录操作日志
     └── 同步失败 → 显示错误详情，记录日志，提供重试按钮
```

## 8. 限制与注意事项

- 模组版本检测基于 Modrinth API，非 Modrinth 来源的模组无法检测更新
- 禁用模组通过重命名实现（`.jar` → `.jar.disabled`），Forge/NeoForge 原生支持此方式
- 文件同步采用单向同步（模板 → 服务器），双向修改时需手动干预
- 大文件传输时显示进度条，避免界面卡死
- Modrinth API 存在请求频率限制（rate limit），批量检测时使用请求队列 + 指数退避策略
- 大模组目录（300+ 模组）下，模组列表采用虚拟滚动优化渲染性能，版本检测采用分批异步查询
- 外部文件变更处理：软件启动时全量扫描；运行期间通过 `fs.watch` 监听目录变更，变更后自动刷新对应列表
- 打包方案：使用 electron-builder + NSIS 安装包，支持自动更新
